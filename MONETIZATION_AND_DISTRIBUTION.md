# SOUNDTEST.PRO 商业化与分发方案记录

本文记录 SOUNDTEST.PRO 后续商业化、付款认证、订阅、买断版和封装下载方案，供后续开发和产品决策使用。

## 产品商业模式

SOUNDTEST.PRO 建议采用“双商业模式”：

- Web/PWA：免费版 + 订阅制。
- 桌面/离线 App：一次性买断下载。
- 企业场景：Team/Cloud 年费订阅。

这样可以覆盖轻度用户、个人专业用户、离线工具用户和企业用户。

## 目标用户群体与 SEO 定位

SOUNDTEST.PRO 的推广定位应覆盖“居民取证入口 + 专业场景工具 + 企业巡检采购”三层用户。文案重点不是承诺法定检测，而是突出实时分贝、地点标注、声学证据照片、录音录像、PDF/CSV 报告和协同留痕。

核心用户群体：

- 居民、业主、租客：楼上噪音、邻里噪音、装修施工、夜间扰民、宠物叫声、广场舞、空调外机噪音。关键词：噪音投诉证据、邻里噪音取证、楼上噪音记录、装修噪音分贝测试、手机测噪音分贝。
- 物业、社区、业委会：住户投诉处理、公共区域巡查、设备房噪音、商铺扰民、社区沟通报告。关键词：物业噪音巡查工具、社区噪音记录、噪声投诉处理报告、小区噪音检测记录。
- 施工、装修、工程团队：施工噪音自查、工地巡检、夜间施工记录、项目现场留痕。关键词：施工噪音监测、工地噪音记录工具、装修噪音分贝标准、现场噪声报告生成。
- 办公室、工厂、门店、连锁企业：办公环境噪音、设备噪音、员工舒适度、门店背景噪声、巡检报告。关键词：办公室噪音检测、工厂噪声巡检、设备噪音记录、门店噪声管理、企业噪音报告。
- 学校、医院、酒店、园区：安静环境管理、投诉留痕、敏感区域噪音控制。关键词：学校噪音监测、医院噪音管理、酒店噪音投诉记录、园区噪音巡检。

## 订阅制方案

订阅适合需要持续服务和服务器成本的功能：

- 云备份。
- 多设备同步。
- 团队协作。
- 用户反馈与报错工单。
- 批量 PDF/CSV 导出。
- 高级证据模板。
- 校准档案管理。
- 企业报告抬头。
- 共享证据库。
- 长期合规资料更新。

建议分层：

- Free：实时分贝、基础记录、单条下载、有广告。
- Pro：免广告、长时录制、批量 PDF、证据模板、校准档案、云备份。
- Team Annual：企业项目空间、5 个成员席位、企业报告抬头、共享证据库、成员权限、审计日志、统一导出、年度云留存、合规资料。

反馈与报错建议分层但不封死：

- Free：提供基础反馈和报错入口，可提交设备、浏览器、复现步骤和诊断信息；严重兼容问题进入公共修复队列。
- Pro：标记为优先支持，正式版进入账号工单系统，适合订阅付款、报告导出、长时录制和校准问题。
- Team Annual：企业优先支持，关联企业项目空间、成员账号和审计日志；正式版建议承诺 1 个工作日内响应企业阻塞问题。

### Team/企业版需求

企业版面向物业公司、检测服务商、连锁门店、施工巡检、园区管理和律师/顾问团队。核心需求不是单个用户多导出几份报告，而是多人协作、证据一致性和可追溯管理：

- 企业项目空间：按小区、门店、项目、客户或案件建立空间，集中管理噪声记录。
- 成员与权限：至少支持 5 个成员席位，区分管理员、采集员、审阅员和只读成员。
- 共享证据库：团队成员上传的录音、录像、声学证据照片、PDF/CSV 报告统一归档。
- 企业报告抬头：统一企业名称、Logo、项目编号、联系人和报告模板。
- 审计日志：记录谁在何时创建、编辑、导出、删除证据，便于内部管理和争议追踪。
- 合规资料：提供统一隐私说明、录音录像提示、数据留存说明和正式检测复核声明。
- 云留存策略：默认年度云留存，支持导出和删除；正式版需提供管理员级数据保留期限配置。
- 支付与发票：支持年度订阅、对公付款、发票信息和续费提醒。

用户自己生成的单条证据不建议完全锁死下载。更合理的付费边界是限制批量、时长、模板、云同步和团队能力。

## 设备默认校准基线

SOUNDTEST.PRO 不应把“普通用户自己调高/调低分贝”作为主流程。同行做法和公开研究更接近以下思路：按设备类型给默认推荐基线，保留高级校准入口，并明确正式证据仍需经检定声级计或声学校准器复核。

调研依据：

- NIOSH 曾测试大量手机声级计应用。结果显示，部分 iOS 应用在内置麦克风上可接近参考声级计，部分场景达到约 ±2 dBA；搭配外接可校准麦克风后可进一步接近 ±1 dB。
- Android 设备因品牌、麦克风、系统音频链路和浏览器实现差异大，公开资料通常不建议给 Android 做统一精确承诺。
- 网页端通过 Web Audio API 得到的是数字音频幅度，需要映射到 dB SPL；桌面浏览器还会受到笔记本/外接麦克风、声卡增益、系统输入音量、自动增益、降噪和回声消除影响，因此 PC 网页读数常见偏高。
- 多数严肃工具会提供“校准”或“外接校准麦克风”建议，而不是允许普通用户随意修改证据数值。

当前建议默认值：

- `desktop-browser`：PC 桌面浏览器，`-12 dB`。用于优先压低 PC 端网页偏高读数。
- `tablet-browser`：平板浏览器，`-8 dB`。作为中间基线。
- `android-browser`：Android 浏览器，`-8 dB`。设备碎片化大，默认只做保守修正。
- `ios-browser`：iPhone / iPad，`-6 dB`。iOS 一致性相对更好，因此修正幅度略小。

产品表达建议：

- 普通用户看到的是“设备推荐校准基线”，不是“随便调整分贝”。
- 手动校准入口必须继续放在高级设置，并提示需要专业声级计或 94 dB / 1 kHz 声学校准器。
- 报告和证据照片必须保留校准偏移、麦克风增益和设备环境，避免证据解释断层。
- 正式商业版可逐步建立设备校准库：按机型、浏览器、系统版本、外接麦克风型号保存更细的基线。

## 付款方案

Web/PWA 端：

- 国际市场优先 Stripe。
- 中国市场可接微信支付和支付宝。
- MVP 阶段可以先只接 Stripe，后续再补中国支付。

iOS App：

- 必须使用 Apple In-App Purchase。
- 不应引导用户绕过苹果内购去网页付款，否则可能被拒审。

Android App：

- Google Play 上架必须使用 Google Play Billing。
- 中国安卓应用市场可按渠道接微信支付、支付宝或应用市场支付。

## 账号与会员认证

正式版本不能只用前端 `localStorage` 判断会员，因为容易被篡改。推荐流程：

```text
用户账号 -> 支付平台 -> 后端验证付款 -> 写入会员权益 -> 前端按权益解锁功能
```

建议支持的登录方式：

- 邮箱验证码登录。
- 手机号验证码登录。
- Google 登录。
- Apple 登录。
- 中国市场可增加微信登录。

最小可行版本可以先做：

- 邮箱验证码登录。
- Stripe Web 订阅。
- 后端会员权益接口。

后端需要保存：

- 用户账号。
- 订阅状态。
- 订阅过期时间。
- 支付平台订单号。
- 权益等级：`free / pro / team`。
- 设备/浏览器登录状态。
- 付款 webhook 记录。

支付成功后，必须由后端接收 Stripe / Apple / Google / 微信 / 支付宝回调，再更新会员状态。

## 地图服务与 Key 管理

普通用户不需要地图 Key，也不应该被要求申请 Google Maps、Amap 或 Baidu Key。地图服务 Key 属于平台运营配置，应由管理员在后台统一维护，并通过后端代理或受限域名 Key 调用，避免前端暴露高权限密钥。

建议策略：

- 海外用户：优先 Google Maps/Google Geocoding；如果只是打开外部地图，可使用 Google Maps URL，不需要用户 Key。
- 中国大陆用户：优先高德/百度；坐标转换和逆地理编码由平台统一处理。
- 公共兜底：未配置商业地图 Key 时，可用 OpenStreetMap/Nominatim 做低频地点识别兜底，但要遵守其使用政策和频率限制。
- 管理员配置：Google Maps Key、高德 Key、百度 Key 均属于超级管理员设置，普通订阅用户不可见。
- 正式生产：建议由后端保存 Key，前端只请求平台接口；Key 应限制域名、接口范围、调用额度和告警。

## 买断版方案

可以买断，而且适合 SOUNDTEST.PRO 的本地工具定位。买断版建议定位为本地离线专业版：

- 一次性购买 App。
- 本地测分贝。
- 本地录音录像。
- 本地导出 PDF/CSV/证据图片。
- 本地校准档案。
- 无广告。
- 不依赖云端账号。

可命名为：

- SOUNDTEST.PRO Pro Lifetime。
- SOUNDTEST.PRO Offline Pro。
- SOUNDTEST.PRO Local License。

买断版应明确只包含“本地软件能力”，不包含云同步、团队协作、长期云服务等持续成本功能。

## 封装方式

当前单页 Web/PWA 可以封装为：

- Windows / macOS / Linux 桌面软件：优先 Tauri，也可 Electron。
- Android App：Capacitor 或 Flutter。
- iOS App：Capacitor 或 Flutter，但买断必须遵守 App Store 付费下载或 Apple IAP 规则。
- PWA 离线版：可安装，但付费验证较弱。

推荐桌面端优先使用 Tauri：

- 体积小。
- 性能好。
- 适合本地工具。
- 可结合本地文件系统、授权文件和离线能力。

## 买断授权方式

可选方案一：本地许可证。

- 用户付款后获得 license key。
- App 离线保存授权。
- 可限制激活设备数量，例如 2 台或 3 台。
- 可定期联网校验，例如 30 天一次。
- 离线仍可使用基础本地功能。

可选方案二：商店买断。

- App Store / Microsoft Store / Mac App Store 直接付费下载。
- 授权由商店处理。
- 用户体验简单。
- 需要接受平台抽成和审核限制。

## 推荐路线

### 阶段 1：Web/PWA 免费版验证

目标是先验证用户是否真的会用 SOUNDTEST.PRO 记录噪声，而不是过早建设完整付费系统。

必须完成：

1. 保留当前 Web/PWA 免费版，确保首次打开即可开始监测。
2. 前端继续模拟 Free / Pro / Team 权益，用于验证价格页、广告位和升级入口。
3. 明确广告位、订阅入口、买断版说明和 Team 咨询入口。
4. 完成核心证据链：实时分贝、时间、地点、照片/录像水印、录音、PDF/CSV 导出。
5. 在设置、报告和证据图片中持续提示：手机测量用于环境记录和投诉辅助，不替代法定检测。

暂不建议做：

- 不急于接入复杂团队后台。
- 不急于做多支付渠道。
- 不把用户自己的单条记录下载强制锁死。
- 不承诺法律效力、监管认证或精密声级计准确度。

阶段 1 的判断标准：

- 用户能在 30 秒内完成一次噪声记录。
- 用户能理解免费版、Pro、Team 和买断版的区别。
- 证据照片和报告看起来足够专业，愿意用于投诉沟通或内部留痕。

### 阶段 2：Pro 订阅闭环

目标是建立最小可行的付费闭环，让个人专业用户为免广告、长时记录、批量导出和更专业模板付费。

必须完成：

1. 增加邮箱验证码登录。
2. 接入 Stripe Web 订阅，先覆盖国际用户和可用信用卡用户。
3. 增加后端会员权益接口，前端只根据后端返回权益展示功能。
4. 前端按权益隐藏广告、解锁 Pro 功能、展示订阅状态和到期时间。
5. 增加支付成功、支付失败、取消订阅、续费失败和恢复订阅的用户提示。
6. 建立基础客服链路：付款问题、报告导出问题、设备校准问题可提交诊断信息。

Pro 版优先解锁：

- 免广告。
- 长时记录。
- 批量 PDF/CSV 导出。
- 高级证据模板。
- 校准档案。
- 云备份或多设备同步的基础版本。

阶段 2 的判断标准：

- 用户完成支付后无需刷新或重新登录即可看到权益变化。
- 订阅状态以后端和支付 webhook 为准。
- 取消订阅后，到期前仍保留权益，到期后自动降级。

阶段 2 详细实施清单：

账号与身份：

1. 建立用户表，至少包含 `user_id`、邮箱、创建时间、最后登录时间、默认语言、地区和删除状态。
2. 首发优先邮箱验证码登录；Google、Apple、微信登录可在支付闭环稳定后再补。
3. 前端登录态只保存短期 session token，不在前端保存“是否会员”的最终判断。
4. 提供退出登录、账号删除申请、云端数据导出和云端数据删除入口。

Stripe 订阅：

1. 后端创建 Stripe Customer，并与内部 `user_id` 绑定。
2. 前端点击 Pro 时向后端请求 Stripe Checkout Session，不直接在前端写价格或密钥。
3. Stripe Checkout Session 需要携带 `user_id`、价格 ID、语言、成功回跳地址和取消回跳地址。
4. 支付成功页只显示“正在同步权益”，最终权益以后端查询结果为准。
5. 提供 Stripe Billing Portal 入口，用于用户自助更新卡片、取消订阅和查看账单。

Webhook 与状态机：

1. 后端必须校验 Stripe webhook signature，不能信任前端回跳参数。
2. 重点处理 `checkout.session.completed`、`customer.subscription.created`、`customer.subscription.updated`、`customer.subscription.deleted`、`invoice.payment_succeeded`、`invoice.payment_failed`、`charge.refunded`。
3. 订阅状态建议统一映射为：`free`、`trialing`、`active`、`past_due`、`canceled`、`expired`。
4. 取消订阅后保留 `current_period_end` 前的 Pro 权益；到期后由后端自动降级。
5. 续费失败进入 `past_due`，前端提示用户更新付款方式，但不立即删除本地证据。
6. 退款或争议订单需要记录审计日志，并按规则撤销后续周期权益。

会员权益接口：

1. 新增 `GET /api/me/entitlements`，返回 `plan`、`status`、`currentPeriodEnd`、`features`、`supportLevel` 和 `source`。
2. 前端每次启动、登录、支付回跳、恢复订阅时重新拉取权益。
3. 前端只根据后端 `features` 显示或隐藏广告、批量导出、模板、校准档案和云备份入口。
4. Free 用户仍可下载自己的单条证据；Pro 主要解锁免广告、批量导出、长时录制、模板和校准档案。
5. 后端应保留权益变更审计：谁、何时、因为什么订单或 webhook 改变了权益。

前端降级与容错：

1. 权益接口不可用时，不应阻断本地监测、录音录像、拍照和单条下载。
2. 付费功能可显示“权益同步中”或“稍后重试”，避免误判为永久失效。
3. 如果本地原型方案与后端权益冲突，正式版以后端权益为准。
4. 广告隐藏、批量导出和模板解锁都应由同一份 `features` 映射控制，避免页面之间不一致。
5. 记录支付、权益同步和降级错误的诊断信息，方便客服处理订阅问题。

### 阶段 3：Team/Cloud 企业版

目标是把 SOUNDTEST.PRO 从个人工具扩展到物业、施工巡检、门店、园区和顾问团队的协作工具。

必须完成：

1. 企业项目空间：按小区、门店、项目、客户或案件集中管理记录。
2. 成员权限：管理员、采集员、审阅员、只读成员。
3. 共享证据库：团队录音、录像、照片、PDF/CSV 统一归档。
4. 企业报告抬头：Logo、项目编号、联系人、统一模板。
5. 审计日志：创建、编辑、导出、删除均可追踪。
6. 年度云留存：支持企业管理员导出、删除和配置留存期限。
7. 企业支付：年度订阅、对公付款、发票信息和续费提醒。

Team 版销售方式：

- 官网放“联系销售/申请试用”入口。
- 可先人工开通企业空间，等需求稳定后再做完整自助购买。
- 对物业、施工和连锁门店准备不同落地页与演示样例报告。

阶段 3 的判断标准：

- 企业管理员能清楚看到成员、项目、证据和导出历史。
- 企业用户愿意把 SOUNDTEST.PRO 作为内部巡检和投诉处理流程的一部分。
- 支持、发票、续费和数据导出流程能被人工兜底。

### 阶段 4：离线买断与应用商店

目标是覆盖不想订阅、需要本地离线工具、或偏好原生 App 的用户。

必须完成：

1. 用 Tauri 封装 Desktop Offline Pro 买断版。
2. 为 Windows / macOS / Linux 提供本地文件保存、离线导出和许可证激活。
3. 用 Capacitor 或 Flutter 准备移动端。
4. App Store 接 Apple IAP 或付费下载。
5. Google Play 接 Google Play Billing。
6. Team/Cloud 仍走企业订阅，不放入永久买断。

正式移动端上架前新增硬性检查：

1. 后端提供账号、权益、云备份、删除/导出数据和服务器签名时间戳，前端 `localStorage` 只能作为本机缓存。
2. 取证编号、本地 SHA-256 和 JSON 证据清单可用于完整性核对，但不能宣传为司法权威时间戳。
3. iOS/Android 必须分别验证锁屏后台录制、低电量、低存储、权限撤销和录制中断恢复。
4. 若宣传 WAV/FLAC 或 24-72 小时值守，必须由原生 App 能力实现并经过真机矩阵验证。
5. 上架材料以 `APP_STORE_READINESS.md` 为准，审核说明必须解释麦克风、摄像头、定位、文件保存和可选后台音频用途。

买断版边界：

- 包含本地监测、本地录音录像、本地导出、本地校准档案和免广告。
- 不包含云备份、团队协作、共享证据库、审计日志和长期云留存。
- 可提供有限期更新，例如 1 年内免费更新，之后大版本升级另行定价。

## 上线前检查清单

### 阶段 1 Web/PWA 公开测试上线口径

当前更适合先发布为 Web/PWA 公开测试版，而不是直接进入应用商店收费上架。阶段 1 的上线目标是验证真实用户是否愿意完成“监测 -> 拍照/录音/录像 -> 保存 -> 导出报告 -> 了解 Pro/Team”的完整路径。

可以上线的前提：

1. 使用 HTTPS 域名部署，不使用普通局域网 HTTP 链接作为对外入口。
2. 首页、首次授权说明、报告、CSV/PDF、证据图片均保留“环境记录与证据辅助，正式检测请以经检定声级计复核为准”的边界说明。
3. Free 版本不依赖登录、支付或管理员功能即可完成核心监测、单条证据保存和基础导出。
4. Pro / Team / Lifetime Offline 只作为权益展示、咨询或模拟，不收真实费用，不承诺云备份或团队空间已经可用。
5. 明确告知录音、录像、照片、定位和历史记录默认保存在本机浏览器；清理浏览器数据可能导致记录或媒体丢失。
6. 至少完成 iPhone Safari、Android Chrome、Windows Edge、macOS Safari/Chrome 的手工冒烟测试，并记录已知限制。
7. 微信内置浏览器、隐私模式、低端 Android 机型和后台锁屏等限制要在帮助文案中提示用户改用系统浏览器或分段保存。

阶段 1 不应该做的事：

- 不接真实 Stripe、Apple IAP、Google Play Billing 或国内支付。
- 不把前端静态 `admin` 门禁宣传为正式后台权限。
- 不把当前读数描述为“执法级”“法定检测”“可直接作为司法鉴定结论”。
- 不承诺录音录像一定能长期保存在浏览器内；应引导用户及时下载到本机文件或相册。

### 正式应用商店上架差距

如果目标是 App Store、Google Play 或国内应用市场正式上架，还需要补齐以下内容：

1. 原生或准原生封装：选择 Capacitor、Flutter、TWA 或原生壳，处理相册/文件保存、后台行为、权限声明和离线资源。
2. 平台支付合规：移动端数字权益必须接 Apple IAP 或 Google Play Billing；不能在 App 内引导绕过平台支付。
3. 后端账号与权限：管理员、会员、企业成员和审计日志必须由后端控制，不能依赖前端 localStorage 或静态密码。
4. 正式隐私政策与用户协议：覆盖麦克风、摄像头、定位、录音录像、PDF/CSV 导出、云备份、账号删除和数据保留期限。
5. 商店素材：应用图标、启动图、截图、功能视频、权限用途说明、年龄分级、隐私标签、支持邮箱和官网。
6. 真机矩阵：至少覆盖 iOS 当前主版本、Android 主流 Chrome/WebView、低内存机型、深色模式、系统语言切换和弱网。
7. 发布与回滚：准备版本号、更新说明、灰度策略、崩溃/错误收集、客服响应和紧急下架/回滚流程。

### 商业化上线前检查

商业化上线前至少确认：

- 免费版可以独立使用，不因付费入口影响核心记录流程。
- Pro 权益说明和实际限制一致，不出现“页面说可用、功能仍锁住”的情况。
- 广告展示不遮挡监测、录音录像、拍照和报告导出等核心操作。
- 订阅付款、取消、续费失败、退款后的权益变化均以后端状态为准。
- 证据照片、PDF 报告和导出文件均包含时间、地点、校准、设备和免责声明。
- 隐私政策明确说明麦克风、摄像头、定位、录音录像、云备份和团队共享的数据用途。
- 地图 Key、支付密钥、云存储密钥和 webhook secret 不出现在前端代码中。
- 提供用户数据导出和删除入口，尤其是云备份和企业空间数据。

## 关键指标

阶段 1 重点看使用意愿：

- 首次监测启动率。
- 完成一次照片/录音/录像证据的比例。
- PDF/CSV/图片导出次数。
- 用户是否点击 Pro、Team 或买断说明。

阶段 2 重点看付费转化：

- Free 到 Pro 的转化率。
- 订阅支付成功率。
- 取消订阅率和续费失败率。
- Pro 功能使用频率，尤其是批量导出、长时记录和模板。

阶段 3 重点看企业价值：

- Team 试用申请数。
- 企业空间内项目数量、成员数量和证据数量。
- 企业导出报告次数。
- 人工支持工单类型和响应时长。

## 主要风险与边界

- 准确度风险：手机和网页测量不能承诺替代检定声级计，所有付费文案都要避免“执法级”“法定检测”等表述。
- 隐私风险：录音、录像、定位和地址属于敏感信息，云功能必须给出清晰授权、删除和导出路径。
- 平台审核风险：iOS 和 Google Play 不允许绕过平台支付购买数字权益，移动端文案和按钮必须按平台规则处理。
- 支付争议风险：会员权益应由后端和支付回调确认，前端模拟权益只能用于 MVP 演示。
- 成本风险：云备份、视频存储、地图逆地理编码和企业留存都有持续成本，必须绑定订阅或企业合同。
- 证据误用风险：报告应定位为“环境记录和沟通辅助”，不要暗示可直接作为官方检测结论。

## 核心原则

- 订阅用于持续服务。
- 买断用于本地离线能力。
- 用户自己的单条证据不应被完全锁死。
- 云同步、团队共享和长期服务不能放入永久买断。
- 各平台支付必须遵守平台规则。
- 会员认证必须以后端为准，前端只做展示和体验控制。
